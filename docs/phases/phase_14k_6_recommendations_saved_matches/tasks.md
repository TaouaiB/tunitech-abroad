# Phase 14K-6 Tasks — Recommendations, Saved Jobs, Match History, Match Detail

## 1. Phase goal

Polish the authenticated candidate intelligence surfaces so recommendations, saved jobs, match history, and match detail feel like one coherent product experience.

This is a **template/UI phase only**. It must improve clarity, hierarchy, actionability, accessibility, French copy, empty states, and responsive layout while preserving all existing backend behavior.

The result should feel like:

- A recommendations page that explains why each job is useful and what the next action is.
- A saved jobs page that is easy to scan and manage.
- A match history page that clearly shows previous checks/results.
- A match detail page that makes score breakdown, missing skills, compatibilities, vigilance points, and next actions readable.
- No new scoring logic, no new recommendation logic, no new LLM calls.

## 5. Hard architecture/privacy rules

- Do not change match scores, score formulas, recommendation ranking, saved jobs logic, or skill gap logic.
- LLM cannot decide final fit score.
- Do not add OpenRouter/LLM calls anywhere.
- Do not add France Travail live API calls.
- User-facing job/recommendation pages must use existing local PostgreSQL-backed data only.
- Public/private object links must use `public_id`; never expose internal integer IDs.
- Do not introduce `<int:...>` routes.
- Do not expose CV files, raw CV text, raw provider payloads, raw LLM text, or secrets.
- Do not use React, Next.js, Vue, Angular, Vite, FastAPI, MongoDB, SQLAlchemy, or SPA architecture.

## 6. Desired UI improvements

### Recommendations

- Use an authenticated app layout consistent with Phase 14K-5 private dashboard.
- Present recommended jobs as scannable cards with:
  - job title/company/location/source fallback
  - score/fit badge if already available in context
  - why it is recommended if already available
  - key missing skills or matched skills if already available
  - save/apply/view actions using existing routes/forms only
- Use proper empty state when no recommendations exist.
- Do not generate fake recommendation explanations if no context exists.
- Do not call LLM.

### Saved jobs

- Present saved jobs as cards/list with clear state and actions.
- Preserve existing save/unsave forms and HTMX behavior if present.
- Use existing job detail routes with UUID `public_id` only.
- Use proper empty state and CTA back to jobs search.

### Match history

- Make past match results readable.
- Use score badge/ring/progress only from existing score values.
- Link to match detail using existing routes only.
- Do not expose internal IDs.
- Add empty state if no history exists.

### Match detail

- Prioritize explanatory hierarchy:
  - score summary
  - compatibilities
  - `Compétences requises manquantes`
  - `Compétences optionnelles à renforcer`
  - `Points de vigilance`
  - action plan / next steps using existing data only
- Preserve existing fields and context exactly.
- Do not change scoring or recompute anything in templates.
- Do not add LLM-generated recommendations.
- Keep quick-match/result forms and links functional.

## 7. CSS strategy

- Prefer existing `.tta-*` classes from Phase 14K-1 and utility fallbacks used in prior 14K phases.
- Add Tailwind utility classes directly in templates only when necessary for layout resilience.
- Do not modify `static/src/css/app.css` unless absolutely necessary for a reusable primitive; if changed, explain why in the report.
- Run `npm run css:build`; `static/css/app.css` may change.

## 8. Required test/check loop

Run after implementation and after every repair:

```bash

python manage.py check --settings=config.settings.local
python manage.py makemigrations --check --dry-run --settings=config.settings.local
python manage.py test --settings=config.settings.local --parallel 1
```


## 9. Repair/enhance loop rule

Gemini and Codex must not stop at the first small issue.

Use:

```text
inspect → implement/verify → fix in-scope issues → enhance within approved docs → test → repeat until PASS/REPAIRED_PASS
```

Do not block for:

- staged/unstaged/mixed working tree
- generated CSS changes
- missing report file
- whitespace
- small copy mismatch
- small template route mistake that can be fixed using an existing route
- accessibility attribute missing if you can add it safely
- a test failure caused by an in-scope template/copy issue you can repair

Block only for:

- product decision needed
- new route/view/model/service needed
- backend logic change needed
- privacy/security risk
- forbidden dependency/stack
- unclear data ownership
- secret exposure
- tests still fail after reasonable in-scope repair
