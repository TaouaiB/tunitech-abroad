# Phase 14K-3 — Public Front Door and Auth

## Goal

Implement the approved UI/UX design for the public front door and authentication pages, without changing backend behavior.

This phase makes the public entry experience look professional and coherent after Phase 14K-1 design-system foundation and Phase 14K-2 global shell/navigation.

## Core scope

Redesign only existing public/home and auth templates.

Expected scope:

1. Homepage / public front door
   - Search-first hero.
   - Clear France-first value proposition.
   - Primary CTA to existing jobs list.
   - Secondary CTA only if it uses an existing real route.
   - Product explanation section anchored as `#comment-ca-marche`.
   - Candidate-focused sections: Tunisian IT profile, France jobs/internships, CV/profile, matching, missing skills.
   - No fake metrics that imply real production usage unless already available in context.

2. Authentication pages
   - Login page.
   - Signup page.
   - Password reset / password change / email confirmation templates if they currently exist and can be safely styled.
   - allauth/socialaccount pages if they currently exist and are part of login/signup flow.
   - Preserve allauth forms, CSRF, provider login tags, hidden inputs, `next` behavior, error rendering, and existing URL names.

3. Optional existing CV-checker page
   - Only style a CV-checker/quick-check public template if it already exists as a working route/template in the current repo.
   - Do not create a new `/cv-checker/` route.
   - Do not invent a public CV upload flow.
   - Do not call matching quick-match globally because it is job-scoped and requires `public_id`.

## Non-negotiable architecture rules

- Public job search must remain local PostgreSQL only.
- No France Travail live API calls from public pages.
- No OpenRouter/LLM calls from views/templates.
- No CV file exposure: never render `cv.file.url`, `upload.file.url`, raw CV text, raw LLM/provider payloads, tokens, or secrets.
- Public URLs must use existing safe URL names and UUID public IDs where object detail routes are involved.
- No internal integer public IDs.
- Templates must not contain business scoring logic.

## UI/UX requirements

- Use the `.tta-*` design-system classes created in Phase 14K-1.
- Preserve the Phase 14K-2 global shell. Do not redesign the base shell again except tiny safe fixes required by this phase.
- Use French UX copy from the approved dictionary.
- Keep adaptive light/dark behavior via system theme.
- Keep mobile-first responsive layout.
- Use accessible headings, labels, focus states, and error messaging.
- Do not use external font/CDN dependencies.
- Do not use placeholder links to non-existing routes.
- Any CTA must resolve to an existing URL name.

## Route safety

Before adding any `{% url %}` tag, inspect current URL names.

Safe examples if they exist:

- `{% url 'core:home' %}`
- `{% url 'jobs:list' %}`
- `{% url 'account_login' %}`
- `{% url 'account_signup' %}`
- `{% url 'privacy:privacy_policy' %}`
- `{% url 'privacy:terms' %}`

Do not use:

- `{% url 'matching:quick_match' %}` without a `public_id`.
- Any invented `/cv-checker/` route.
- Any `/dashboard/settings/` route in public/auth pages.

## Required implementation behavior

Agents must not stop at the first minor issue.

Required loop:

1. Inspect relevant files and current routes.
2. Implement within scope.
3. Run checks.
4. If a check fails due to an in-scope template/copy/style issue, fix it.
5. Re-run failed checks and any affected checks.
6. Repeat until PASS/REPAIRED_PASS or until a true blocker exists.

## Block only for real blockers

Block only if:

- required fix needs backend route/view/model/service change
- required fix needs a new dependency
- required fix needs product decision
- forbidden stack appears
- privacy/security exposure appears
- tests still fail after reasonable in-scope repair
- design documentation conflicts with actual routes and no safe existing route exists

Do not block for:

- staged vs unstaged files
- mixed working tree
- generated `static/css/app.css` changes after CSS build
- missing report file
- trailing whitespace
- small copy/test mismatch that can be fixed safely
