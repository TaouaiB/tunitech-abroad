# Phase 14K-5 Tasks — Candidate Dashboard, Profile, CV, Account

## 1. Phase goal

Polish the authenticated candidate workspace pages so they look like a coherent product instead of raw forms/cards.

Use the approved TuniTech Abroad design system and UI/UX documentation. This is a **template/UI phase only**.

The result should feel like:

- A clear private dashboard/action center.
- A profile page that is easier to complete and review.
- A CV page that clearly explains privacy, status, parsing, replacement, and safe deletion/management actions already available.
- An account/security page that is clear and safe.
- Email preferences styled consistently if the existing route/template exists.

## 5. Hard architecture/privacy rules

- CV files are private.
- Never expose `file.url`, `cv.file.url`, `upload.file.url`, raw CV text, raw provider request/response JSON, or raw LLM text in templates.
- Do not add a CV download link unless an existing secure view already exists and is explicitly present in the current route/template context.
- Do not invent routes or URLs.
- Public and private object links must use `public_id` if a URL needs an object ID.
- Do not introduce `<int:...>` routes.
- Do not use OpenRouter/LLM from templates/views.
- Do not use France Travail live API calls.
- Do not alter matching scores, CV parsing, profile completion calculations, recommendation logic, saved jobs logic, or email preference logic.
- Do not use React/Next/Vue/Angular/Vite SPA code.

## 6. Desired UI improvements

### Dashboard

- Convert into an action-oriented private dashboard.
- Show a concise welcome/next-step hero.
- Keep existing real metrics/status values only.
- Use cards for:
  - profile completion/status
  - CV status/last upload/parse status
  - recommendations entry point if existing context links exist, but do not redesign recommendations page
  - saved jobs entry point if existing route exists, but do not redesign saved jobs page
  - account/security entry point
- Add clear empty states if profile/CV/recommendations are missing.
- Keep links to existing routes only.

### Profile

- Make long forms readable using sections/cards/grids.
- Preserve all form fields, field names, error display, CSRF, method, hidden inputs, and submit behavior.
- Add sticky save bar only if it does not break existing forms.
- Use French labels/help text only at template display level; do not change backend forms.
- Do not remove fields.
- Do not change validation behavior.

### CV page

- Emphasize privacy and status clearly.
- Use dropzone/card UI for existing upload form.
- Preserve multipart form behavior and CSRF.
- Preserve parse status, errors, and replacement/delete actions if they exist.
- Make parsed summary readable only if existing page already shows safe summary fields.
- Do not expose raw extracted text.
- Do not expose CV file URL.
- Do not create fake download/reparse/delete behavior.

### Account/security

- Clear account card, authentication providers, password/security actions, danger zone if already present.
- Preserve allauth/account URLs and forms.
- Do not change delete-account behavior.
- Do not invent security promises.

### Email preferences

- Style existing form/page only if present.
- Preserve unsubscribe/preference semantics and CSRF.
- Do not change email delivery logic.

## 7. CSS strategy

- Prefer existing `.tta-*` classes from Phase 14K-1.
- Add Tailwind utility classes directly in templates only when necessary for layout resilience.
- Do not modify `static/src/css/app.css` in this phase unless absolutely necessary for a missing reusable primitive that is used across these private pages. If changed, explain why in the report.
- Run `npm run css:build`; `static/css/app.css` may change.

## 8. Required test/check loop

Run after implementation and after every repair:

```bash
source .venv/bin/activate

python manage.py check --settings=config.settings.local
python manage.py makemigrations --check --dry-run --settings=config.settings.local
python manage.py test --settings=config.settings.local --parallel 1
npm run css:build
python manage.py collectstatic --dry-run --noinput --settings=config.settings.local
git diff --check
git diff --cached --check || true
```

Run security/architecture greps:

```bash
grep -R "cdn.tailwindcss.com" templates config static -n || true
grep -R "React\|Next.js\|next/\|createRoot\|vue\|angular\|Vite" package.json templates static apps -n 2>/dev/null || true
grep -R "FranceTravailClient\|search_offers\|api.francetravail" apps/*/admin.py apps/*/views.py templates -n 2>/dev/null || true
grep -R "OpenRouterClient\|OPENROUTER\|run_llm\|client.chat\|_make_request" apps/*/admin.py apps/*/views.py templates -n 2>/dev/null || true
grep -R "file.url\|cv.file.url\|upload.file.url\|raw_text\|raw_request_json\|raw_response_text\|raw_response_json" templates apps -n 2>/dev/null || true
grep -R "<int:" apps templates config -n 2>/dev/null || true
git diff -- . ':!.env' ':!.env.*' | grep -iE "sk-or-|Bearer [A-Za-z0-9_.-]{20,}|OPENROUTER_API_KEY=.*[^=]|FRANCE_TRAVAIL_CLIENT_SECRET=.*[^=]|EMAIL_HOST_PASSWORD=.*[^=]" || echo "OK: no obvious secrets in diff"
```

## 9. Repair/enhance loop rule

Gemini and Codex must not stop at the first small issue.

Use:

```text
inspect → implement/verify → fix in-scope issues → enhance within approved docs → test → repeat until PASS/REPAIRED_PASS
```

Do not block for:

- staged/unstaged/mixed working tree
- generated CSS changes
- missing report file
- whitespace
- small copy mismatch
- small template route mistake that can be fixed using an existing route
- accessibility attribute missing if you can add it safely
- a test failure caused by an in-scope template/copy issue you can repair

Block only for:

- product decision needed
- new route/view/model/service needed
- backend logic change needed
- privacy/security risk
- forbidden dependency/stack
- unclear data ownership
- secret exposure
- tests still fail after reasonable in-scope repair
